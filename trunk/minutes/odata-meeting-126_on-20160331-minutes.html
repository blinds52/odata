7. V4.01 issues - Issues for V4.01_WD01 in New or Open state

7.1 OData protocol

7.1.1 ODATA-480 - $batch: remove restriction "within the same change set " for Content-ID referencing

Ram: ODATA-480 is OPEN, no objections

Ralf: Remove restriction "within the same change set " for Content-ID referencing and allow referencing requests within the same batch that appear before the reference and thus already have been processed.

Add term BatchContentIdReferencingAcrossChangeSetsSupported to Capabilities vocabulary, similar to the existing term BatchContentIdReferencingInRequestBodiesSupported.

Ralf: (cf. ODATA-541 - Allow the If-Match header to reference the ETag that resulted from an earlier operation in a batch request)

Ralf: I move we resolve ODATA-480 as proposed. Mike seconds.

Motion passes, no objections. ODATA-480 is RESOLVED as proposed.

7.1.2 ODATA-921 - Specify operations for keyless nav props

Ram: ODATA-921 is OPEN, no objections

Mike: if you have a navigation property with no navigation property binding, the canonical URL may not uniquely identify a members

Mike: Revised proposal for clarification:

Mike: Clarify:
-Can't have containment nav props of abstract types w/o keys (including Edm.EntityType)
-add note to url conventions: if you have a navigation property with no navigation property binding, the canonical URL (formed by appending the key of the contained member to the navigation property path) may not uniquely identify a members

Mike: slight modification:

Mike: Clarify:
-Can't have containment nav props of abstract types w/o keys (including Edm.EntityType)
-add note to url conventions: if you have a navigation property with no navigation property binding, the canonical URL (formed by appending the key of the child member to the navigation property path) may not uniquely identify a members.

Mike: I move we resolve ODATA-921 as proposed. Mark seconds.

Motion passes, no objections. ODATA-921 is RESOLVED as proposed.
