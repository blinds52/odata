6. V4.01

6.1 Issues for V4.01_CSD02 ready for resolution

6.1.1 ODATA-1028 - 4.3.4 and 4.4: clarify resolution of relative URLs in system query option $id

Ralf: Just realized that 4.3.4 contradicts 4.4, http://docs.oasis-open.org/odata/odata/v4.01/csprd01/part2-url-conventions/odata-v4.01-csprd01-part2-url-conventions.html#_Toc470093604:

The entity-id specified by $id may be expressed absolute or relative to the request URL.

So we have two different resolution mechanisms for the $id system query option:
- relative to the service root for paths containing /$entity
- relative to the request URL for paths ending in /$ref

This is less than consistent.

Ralf: Source of problem:
http://host/service/$entity?$id=Products(0)
http://host/service/$entity/this.VipProducts?$id=Products(0)
http://host/service/$entity/this.VipProducts?$id=../Products(0)
DELETE http://host/service/Categories(1)/Products/$ref?$id=../../Products(0)
DELETE http://host/service/Categories(1)/Products/$ref?$id=/service/Products(0)

Mike: could also do http://host/service/$entity/this.VipProducts?$id=/service/Products(0)

Hubert: I move to resolve ODATA-1028 as proposed.

Ralf: ... in this discussion, contrary to the proposal in the issue

Mike: updated proposal in issue: Clarify that relative URLs passed with query option $id are relative to the path to be consistent with $ref.

Hubert: I move to resolve ODATA-1028, once again, as per the updated proposal. Martin seconds.

Ralf: ODATA-1028 is resolved again

6.2 Issues for V4.01_CSD02 in New or Open state without concrete proposal

6.2.1 ODATA-920 - Specify overflow for int data types (-INF, INF, NaN)

Mike: Proposal:
1) Because annotating numeric values of arrays is problematic, and we are no longer trying to use JSON Schema as our schema language, revert application of the numericValueException annotation and continue the 4.0 literal "INF", "-INF", and "NaN" values.
2) Support these values all numeric types.
3) Support -INF and INF for Edm.Date and Edm.DateTimeOffset --> OData Temporal Extension

Mike: I move we resolve ODATA-920 as proposed. Mark

Ralf: ODATA-920 is resolved with the revised proposal
