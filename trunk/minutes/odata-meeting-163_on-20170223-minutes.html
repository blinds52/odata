6.1.1 ODATA-1035 - 10.13: Simplify context URL for Property Value

Ralf: The context URL for a property value contains the keys of the entity.
This seems overly specific as the context URL is a pointer to structural metadata, so why should it vary from entity to structurally identical entity?
Note that the context URL for an entity does not contain the key, nor is it required that the key properties are always present in the response.
Example: http://host/service/$metadata#Customers(1)/Addresses
Allow the context URL pattern 

    {context-url}#{type-name}{select-list}

also for property values.

If something more specific is desired, the pattern could be 

    {context-url}#{entity-set}/{property-path}{select-list}

Ralf: Example: http://host/service/$metadata#Customers/Addresses
Example: http://host/service/$metadata#Customers/Addresses(Street,Zip)
Example for operation: http://host/service/$metadata#Collection(self.Address)

Mark: Tried it out with his implementation: it is already using the key-less pattern
"@odata.context": "http://localhost:8080/api/odata4/sforce_odata/$metadata#USERS/USERNAME"

Ralf: ODATA-1035 is OPEN

Martin: Should the pattern with key still be allowed?

Ralf: The ISO version of OData V4 allows this, so disallowing it is problematic

Ramesh: I move ODATA-1035 resolved as proposed. Martin seconds.

Ralf: ODATA-1035 is RESOLVED as proposed

6.1.2 ODATA-1036 - 10.7/8: Simplify context URL for "all operations in schema"

Ralf: Background: $select=self.*

Ralf: Context URL: #WorkflowItems(self.Approve,self.Reject,self.Forward)

Ralf: Proposal: Context URL: #WorkflowItems(self.*)

Ralf: Precedence - structural properties: $select=Name,Date

Ralf: Context URL: #WorkflowItems(Name,Date)

Ralf: Precedence - structural properties: $select=*

Ralf: Context URL: #WorkflowItems(*)

Ralf: ODATA-1036 is OPEN

Matt: Likes it

Mark: I move that ODATA-1036 be accepted as proposedMatt seconds.

Ralf: ODATA-1036 is RESOLVED as proposed

6.1.3 ODATA-1037 - Add tagging term Core.Deprecated to mark deprecated model elements

Ralf: Our model versioning rules only allow adding optional model elements, they don't allow removing model elements.

One way to deal with this is to leave outdated model elements in a service and just mark them as "deprecated".

Ralf: Add new Core term 

 
  <Term Name="Deprecated" Type="Core.Tag" DefaultValue="true"> 
    <Annotation Term="Core.Description" String="Model elements annotated with this term are deprecated and should no longer be used. They may be removed in a future version of the service." /> 
  </Term>

Mark: Seems to make sense as a general concept

Ramesh: Agrees

Ralf: ODATA-1037 is OPEN

Ramesh: I propose to resolve ODATA-1037 as proposed. Martin seconds.

Ralf: ODATA-1037 is RESOLVED as proposed
