5.4 ODATA-1109 - Clarify that 'parameter' aliases can also be used to substitute expressions

Mike: Draft Proposal: Query parameters for $filter segment are always expressions. All other parameters are treated as literals.

Hubert: Calls out that the last example (taking the input 1&$expand=Orders) is a general issue. We should provide general guidance that, when taking input from a user, must make sure it is the expected type; for example, make sure there are no unescaped '&'.

Mike: Clarified proposal: Query parameters for $filter segment are always passed in as expressions. All other parameters are evaluated and passed as literals.

Mike: Added new issue for Securing OData: https://issues.oasis-open.org/browse/ODATA-1110

George: I'm wondering whether we could insist that the segment be a complete expression, rather than 'part' of an expression.

Mike: Issue:

Customers?$filter=Orders($filter=ShipTo eq @a)&@a=City

Hubert: Can one do this:

Customers?$filter=Orders($filter=ShipTo eq @a;@a=City)

George: Can one do this:

Customers?$filter=Orders($filter=ShipTo eq City)

Ralf:

Customers(ID=@c)/Orders(ID=@o)?@c=ID&@o=45

Mike:

Customers/GetById(id=@id)?@id=1

Ralf: @Hubert: no, nested option syntax with semicolons doesn't allow nested aliases

Mike:

Customers?$fitler=id eq @id?@id=1
Customers?$filter=id eq @id&@id=1
Customers/GetById(id=@id)?@id=Manager/BestEmployee()/Id

Mike: (that above example was invalid...)

Ralf: Parameter aliases allow the same value to be used multiple times in a request and may be used to reference primitive values, complex, or collection values.

products/ByColors(colors=@c)?@c=["red',"green","blue"]
products/ByColors(colors=@c)?@c=$root/My/Preferences/Colors

Mike:

Customers?$filter=Orders($filter=ShipTo eq @a)&@a=City

Ralf:

Customers(ID=@c)/Orders(ID=@o)?@c=ID&@o=45
Customers(ID=@c)/Orders(ID=@c)?@c=FirstName&@o=45
Customers(ID=@c)/Orders(ID=@o)?@c=FirstName&@o=45
Customers(ID=42)/some.function(par1=FirstName)/...
Customers(ID=42)/some.function(par1=@p)/...?@p=FirstName

Hubert, Ralf: tend to have expression evaluated at point where it is used. In last example evaluation should not depend on subsequent path segments ...

Mike: tends to have expression evaluated in the context of the last path segment
