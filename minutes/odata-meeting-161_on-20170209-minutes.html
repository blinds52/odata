6.1.3 ODATA-1030 - Consider support of NavigationPropertyBindings to non-contained nav props

Ralf: ODATA-1030 is OPEN

Ralf: Instead of adding a NavigationPropertyConstraint XML element add a term in the Validation vocabulary

Mike: do the same for ODATA-674?

Hubert: is in favor of using the existing XML element for ODATA-674 and allow it in a new position

Hubert: there might be other constraints on navigation properties in the future, so resolving ODATA-674 via an annotation would be more consistent in the long term

Stefan: Is the constraint on the Navigation or on navigation properties?

Mike: It's actually defined on the type containing the nav prop and is a constraint of the membership of the nav prop.

Mike: Not sure if that helps...

Mike: Proposal: Apply ODATA-674 as an annotation term in the Validation vocabulary "MemberOf" that can be applied multiple times to the same path and can target any collection-valued navigation property.

Mike: How would we differentiate multiple applications to the same target?

Mike: Or even multiple targets?

@MemberOf:[
  {path,target},
  {path,target}
]

Ralf: use @ItemOf

Stefan: Names in JSON for the "members" of "objects"; from RFC 7159

object = begin-object [ member *( value-separator member ) ]
               end-object
member = string name-separator value

Mike: Revised proposal:
Apply ODATA-674 as an annotation term in the Validation vocabulary "ItemsOf". The value of the ItemsOf annotation is a collection of complex types containing two path properties; "path" and "target". The same path can occur multiple times and the target can be any collection-valued navigation property. If the target is a containment navigation property then the constraint defines where the entity lives.

Hubert: I move to resolve ODATA-1030 as per the revised proposal. Stefan seconds.

Ralf: ODATA-1030 is resolved with the revised proposal
